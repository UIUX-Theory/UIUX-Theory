# 코칭 플레이북 (확장 자료)

> SKILL.md의 프로토콜로 부족할 때, 또는 사용자가 막혔을 때 펼친다.

---

## 사용자가 자주 놓치는 큰 그림 — 체크리스트

단편적 사고에 머물러 있다고 판단되면, 아래 중 *빠진 관점 하나*를 골라 짚어준다. (한 번에 다 던지지 않는다)

- **퍼널 관점**: "지금 보시는 건 화면인데, 이 화면이 퍼널의 어느 단계인가요?"
- **목표 지표**: "이 변경의 성공·실패는 어떤 숫자로 판정되나요?"
- **사용자 세그먼트**: "타겟이 신규/기존/충성 중 누구인가요? 다 다른 행동을 합니다."
- **비교 대상**: "사용자가 이 화면 *안에서* 비교하나요, *다른 앱과* 비교하나요?"
- **인지 부하**: "한 화면에 의사결정 포인트가 몇 개인가요? 4개 이상이면 위험."
- **첫 진입 기대**: "처음 들어온 사용자가 3초 안에 무엇을 알아야 하나요?"
- **이탈 지점**: "어디서 이탈 가능성이 가장 높나요? 거기가 진짜 개선 포인트입니다."
- **묶음의 단위**: "지금 다루는 게 단일 화면인가요, 컴포넌트인가요, 플로우인가요?"
- **반복 vs 일회성**: "1회 사용 vs 반복 사용 — 학습곡선을 어디까지 허용할까요?"

## 막혔을 때 던질 질문 카드 8개

1. "이 요건은 어떤 문제를 해결하기 위한 건가요? 한 문장으로요."
2. "이 변경의 성공을 판정할 숫자 하나만 고른다면 무엇인가요?"
3. "기획자가 가져온 솔루션 말고, 같은 문제를 푸는 다른 방법 3개를 떠올려 볼까요?"
4. "데이터로 확인 가능한 부분과 추측인 부분을 나눠 볼까요?"
5. "5개 요건 중 임팩트가 가장 큰 1개만 한다면? 나머지는 왜 후순위인가요?"
6. "이 화면이 퍼널의 어디인가요? 앞 단계와 뒷 단계는 어디예요?"
7. "처음 진입한 사용자가 3초 안에 알아야 할 정보는 무엇인가요?"
8. "이 변경이 다른 어떤 화면·기능에 영향을 주나요?"

## 안티패턴 — 절대 하지 말 것

- ❌ 요건 5개를 "어떻게 다 충족시킬까"부터 고민하기
- ❌ "왜 필요한가" 없이 "어떻게 만들까"로 직행하기
- ❌ 모든 요건에 동일한 디자인 노력 배분하기
- ❌ 데이터 없는 추측을 "사용자는 이럴 거예요"로 단정하기
- ❌ 단일 화면 시야로 시스템·퍼널 영향 무시하기
- ❌ "예쁘게/깔끔하게/직관적으로" 같은 모호어로 결정 종료하기
- ❌ *솔루션*을 두고 토론하기 (→ 문제로 되돌려야 한다)
- ❌ "기획에서 그렇게 정해졌어요"로 사고 멈추기

## Bad → Good 실제 사례

**Bad — 기획자 요건 5개에 대한 단편 코멘트**
> "정보 축약 필요" / "UI 변경 또는 팝업 대체할 순 없는지" / "검색 GA 데이터 확인 필요" / "안으로 필터를 숨기는 것인지" / "검색 사용자가 많으면 해당 UI 축소가 사용성을 저해할 수도"

→ 각 요건에 1:1 코멘트가 붙었을 뿐, **상위 목표가 보이지 않음.**

**Good — 동일 화면을 퍼널로 본 구조**
> 문제정의(3) → 가설(3) → **Goal: 진입고객 → 최종 매수 전환률 개선(퍼널)** → **Sprint: 상품 리스트 → 상품 상세 전환률**

→ 5개 요건이 아니라 **단 하나의 지표로 수렴.** 모든 디자인 결정이 그 지표에 종속됨.

> **이 코칭의 본질은 Bad에서 Good으로 가는 사고 회로를 *사전에* 돌게 하는 것이다.**
