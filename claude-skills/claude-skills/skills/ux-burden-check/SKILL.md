# UX 인지 부담 진단 (UX Burden Check)

바이브 코딩으로 만든 UI를 **심리학 렌즈로 진단**하고, 사용자에게 확인받은 뒤 코드를 고치는 스킬입니다.

목표는 하나예요 — **사용자가 덜 생각하게 만드는 것.**
UI 자체가 아니라 **인지 부담의 위치**를 찾습니다. 화면 어디에서 사용자가 무엇을 외우고, 계산·비교하고, 조작하고, 불안해하는지를 봅니다.

> 이 스킬은 *수치 디테일*(여백·간격·대비·터치영역)을 다루지 않아요. 그건 `ui-detail-check`의 영역입니다.
> 여기서 다루는 건 **왜 사용자가 멈추는가** — 멘탈 모델, 망설임, 편향, 감정 부담이에요.

---

## 동작 흐름 (반드시 이 순서)

### 1단계 — 진단 (Diagnose)

화면(또는 방금 만든 UI 코드)에서 사용자가 멈추거나 헷갈릴 지점을 찾습니다.
각 상호작용 지점마다 **부담의 4유형** 중 무엇인지 먼저 분류하세요:

1. **기억 부담** — 사용자가 이전 화면/단계의 무언가를 *외우고* 있는가?
2. **결정 부담** — 비교하고 *선택*해야 하는가? 신호 대 잡음은 괜찮은가?
3. **운동 부담** — 도달하고 *조작*해야 하는가? (손의 출발점·도달 거리)
4. **감정 부담** — *불안·의심·책임감·자책*을 느끼는가?

> UI 문제처럼 보이는 것의 대부분은 위 네 부담 중 하나의 **위치 문제**입니다.

망설임의 *순간*을 더 정확히 짚으려면 사용자 독백을 추정하세요:
- "뭘 해야 되지?" / "이렇게 하는 게 맞나?" → **실행의 간극** (시그니파이어·라벨을 본다)
- "된 건가?" / "내가 뭘 한 거지?" → **평가의 간극** (피드백·상태 표시를 본다)

추천·검색 결과, 가격·결제, 옵션 선택, 동의·알림 화면이면 **인지 편향**도 작동하는지 확인하세요 (손실 회피·앵커링·현상 유지). 이때는 *다크 패턴 경계*를 함께 봅니다.

판단 근거가 더 필요하면 `references/`를 읽으세요 (아래 [레퍼런스 사용](#레퍼런스-사용) 참고).

### 2단계 — 보고 (Report)

발견한 부담을 **구체적으로, 우선순위 순으로** 정리합니다. 모든 법칙을 나열하지 마세요.
각 항목은 이렇게:

- **어떤 요소가** (코드/화면의 어느 부분)
- **어떤 부담을** (기억/결정/운동/감정 중 무엇)
- **왜** (사용자가 그 지점에서 무엇을 생각하게 되는지)

법칙명은 *근거가 필요할 때만* 가볍게 덧붙이고, 강의하지 않습니다.

### 3단계 — 확인 (Confirm) — **코드를 고치기 전 반드시 멈춘다**

진단을 보여준 뒤, **무엇을 고칠지 사용자에게 확인받습니다.**
바로 코드를 건드리지 마세요. 예:

> "1번(기억 부담)과 3번(감정 부담)이 우선 같이 보여요. 어느 쪽부터 고칠까요? 아니면 전부?"

사용자가 고를 항목을 정하면 다음 단계로 갑니다.

### 4단계 — 수정 (Fix)

확인받은 항목**만** 코드에 반영합니다.
- 부담을 *어디로 옮기는지* 의식하세요 — 화면에서 줄인 복잡성이 사용자 머릿속이나 다음 화면으로 전가되지 않는지.
- 수정 후, 그 변화로 **사용자가 무엇을 덜 생각하게 되는지** 한 줄로 짚어주세요.

---

## 톤 & 스타일

- 존댓말 ~요체, 공손하지만 단정하게.
- 진단은 구체적으로, 보고는 짧게.
- 확인 단계 없이 코드를 고치지 않습니다 (사용자가 명시적으로 "바로 고쳐줘"라고 한 경우는 예외).
- 법칙명을 불필요하게 인용하거나 강의하지 않습니다.

## 레퍼런스 사용

`references/`의 4개 문서는 진단의 *근거 자료*입니다. 필요한 것만 펼쳐 읽으세요.

| 파일 | 언제 읽는가 |
|---|---|
| `references/01_ux_10laws.md` | 부담 유형을 더 깊게 짚을 때. 제이콥·피츠·힉·밀러·포스텔·피크엔드·심미·폰레스토프·테슬러·도허티 + **법칙 충돌 매트릭스** + **부담 4유형 진단 도구**(부록 2) |
| `references/02_nielsen_heuristics.md` | "이건 UI 문제가 아니라 어떤 부담의 문제인가?"를 체크리스트로 짚을 때. 10개 휴리스틱이 각각 부담 유형에 매핑돼 있음 |
| `references/03_norman_gulfs.md` | "왜 사용자가 멈췄는가"를 분류할 때. 실행/평가의 두 간극 + 어포던스·시그니파이어 + **간극 진단 표** |
| `references/04_cognitive_biases.md` | 추천·가격·결제·옵션·동의 화면일 때. 손실 회피·앵커링·현상 유지 + **다크 패턴 경계 표** |

**충돌이 보이면** (예: 익숙함 vs 차별화, 선택지 줄이기 vs 복잡성 전가) 01의 충돌 매트릭스를 근거로 "무엇을 지키고 무엇을 희생하는지" 트레이드오프를 사용자에게 보여주고 고르게 하세요.

## 자가 점검 (수정 후)

- [ ] 부담의 *위치*를 짚었는가 (단순히 "버튼이 문제"가 아니라)?
- [ ] 코드를 고치기 전에 사용자 확인을 받았는가?
- [ ] 줄인 복잡성이 다른 곳으로 전가되지 않았는가?
- [ ] 수정 후 "사용자가 무엇을 덜 생각하게 되는지" 짚었는가?
- [ ] 법칙명을 불필요하게 강의하지 않았는가?
- [ ] 수치 디테일(여백·대비 등)로 새어나가지 않았는가 (그건 ui-detail-check 영역)?
